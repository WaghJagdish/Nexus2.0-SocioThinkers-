Production-Grade Multi-Agent 
Agricultural Advisory System: Backend 
Architecture and Implementation 
The deployment of artificial intelligence within the government technology (GovTech) sector 
requires architectures that balance the probabilistic flexibility of Large Language Models (LLMs) 
with the strict, deterministic boundaries of public policy. The following report details the 
comprehensive backend implementation of a production-grade Agricultural Advisory System. 
This system leverages a multi-agent architecture powered by LangGraph, utilizing Sarvam AI 
for highly accurate, localized voice processing, and a hybrid eligibility engine to facilitate 
automated scheme discovery and form fulfillment. The architecture addresses the critical 
requirements of offline-first potential, multi-lingual support, and verifiable data collection. 
Part 1: Scheme Data Sources and Ingestion Pipeline 
The foundation of any advisory agent is its knowledge base. To ensure high-fidelity responses, 
the system must continually ingest, structure, and update data from authoritative government 
platforms. The fragmented landscape of Indian agricultural data requires integration across 
central platforms like the myScheme portal, India.gov.in, the PM-Kisan official site, and 
state-level portals such as Maharashtra's MahaDBT.1 
Data Access and Scraping Strategies 
The Indian government has increasingly adopted Open API policies through platforms like API 
Setu, which facilitates interoperable data exchange across e-Governance silos.4 The 
myScheme portal and various agricultural datasets are accessible via API Setu, providing 
structured JSON payloads containing scheme metadata.5 However, for portals lacking robust 
API endpoints, such as specific state-level agricultural mechanization schemes or localized 
subsidy programs, headless browser automation (e.g., Selenium or Playwright) combined with 
asynchronous HTTP clients must be deployed to extract DOM elements.7 
The data extracted from these diverse sources must be standardized into a unified schema to 
facilitate deterministic filtering and semantic search. Table 1 illustrates the normalized structure 
required for the system to process both central schemes (like PM-Kisan) and state schemes 
(like Maharashtra's Tractor Subsidy).

Part 2: RAG and Vector Retrieval System 
To ensure the LangGraph agent can dynamically retrieve relevant schemes based on a user's 
conversational input, a Retrieval-Augmented Generation (RAG) system is strictly necessary. For 
GovTech applications, exact metadata filtering is equally as important as semantic similarity.8 If 
a farmer resides in Maharashtra, retrieving a subsidy scheme exclusive to Punjab—regardless of 
semantic similarity—is a critical system failure. 
PostgreSQL, augmented with the pgvector extension, provides a robust infrastructure for 
combining exact relational filtering with highly dimensional vector similarity search.8 This 
dual-capability ensures that the vector search space is pre-filtered by hard demographic 
realities before the distance metric (e.g., cosine similarity) is calculated. 
Embedding Generation and Vector Storage Implementation 
The retrieval service chunks the comprehensive JSON data into semantic blocks, generates 
embeddings, and executes filtered similarity searches.

Part 3: The Hybrid Eligibility Engine 
Relying exclusively on generative LLMs for determining eligibility introduces unacceptable risks 
of hallucination, non-determinism, and non-compliance with strict government guidelines.10 A 
purely rule-based system, conversely, struggles with the unstructured, fuzzy nature of 
conversational inputs (e.g., a farmer stating, "I have about three acres near the river").12 
The definitive solution is a Hybrid Eligibility Engine following the "Rules-First, LLM-Fallback" 
pattern.10 Deterministic rules execute instantaneously and provide auditable guarantees. When 
an input fails to match predefined structural criteria, the system defers to the LLM to interpret 
the semantic intent before updating the state.10 
Architectural Pattern and Execution Flow 
1. Pattern Matching Phase: Incoming user data is matched against strict regex, numerical 
thresholds, and dictionary lookups.10 
2. Deterministic Evaluation: If the data satisfies the rule completely, the engine 
immediately yields a decision, bypassing the LLM entirely (saving latency and token 
costs). 
3. Semantic Fallback: If the rule evaluation encounters ambiguity (e.g., unparseable text 
formats for land area), a skip_llm flag is toggled to False, invoking the LLM to contextually 
deduce eligibility.10

# services/eligibility_engine.py 
import json 
import logging 
from typing import Dict, Any, Tuple, List 
from langchain_openai import ChatOpenAI 
from langchain_core.prompts import PromptTemplate 
 
logger = logging.getLogger(__name__) 
 
class HybridEligibilityEngine: 
    """ 
    Evaluates farmer eligibility using a deterministic forward-chaining rule  
    evaluation, falling back to an LLM for fuzzy logic resolution. 
    """ 
    def __init__(self): 
        # Temperature 0 is critical for predictable JSON generation in fallback 
        self.llm = ChatOpenAI(model="gpt-4o", temperature=0) 
         
    def _evaluate_deterministic_rules( 
        self,  
        user_data: Dict[str, Any],  
        scheme_rules: Dict[str, Any] 
    ) -> Tuple[bool, bool, List[str], str, float]: 
        """ 
        Evaluates hard business logic based on defined JSON schemas. 
        Returns: (handled_by_rules, is_eligible, missing_fields, reason, confidence_score) 
        """ 
        missing_fields = 
        required_fields = scheme_rules.get("required_fields",) 
         
        # 1. Verification of Required Fields 
        for field in required_fields: 
            if field not in user_data or user_data[field] in [None, "", "unknown"]: 
                missing_fields.append(field) 
                 
        if missing_fields: 
            # Deterministic failure due to missing data. High confidence. 
            return True, False, missing_fields, f"Missing required fields: {', '.join(missing_fields)}", 1.0 
 
        # 2. Numeric Threshold Evaluation (e.g., Land Size) 
        max_land = scheme_rules.get("eligibility_criteria", {}).get("max_land_size_hectares") 
        if max_land and "land_area" in user_data: 
            try: 
                # Attempt strict type conversion 
                user_land = float(user_data["land_area"]) 
                if user_land > max_land: 
                    return True, False,, f"Land size {user_land} ha exceeds limit of {max_land} ha.", 1.0 
            except (ValueError, TypeError): 
                # Value is fuzzy (e.g., "around 2 acres"). Hand control to LLM fallback. 
                return False, False,, "", 0.0 
 
        # 3. Categorical Exclusions (e.g., Pensioners, Institutional Landholders) 
        exclusions = scheme_rules.get("eligibility_criteria", {}).get("exclusions",) 
        user_category = user_data.get("farmer_category", "") 
        if user_category in exclusions: 
            return True, False,, f"Ineligible: User falls under exclusion category '{user_category}'.", 1.0 
 
        # If data is perfectly structured and clears all hurdles 
        return True, True,, "User meets all deterministic criteria.", 1.0 
 
    async def _evaluate_llm_fallback( 
        self,  
        user_data: Dict[str, Any],  
        scheme_rules: Dict[str, Any] 
    ) -> Dict[str, Any]: 
        """ 
        Uses an LLM to interpret fuzzy conditions, acting as the cognitive fallback. 
        """ 
        prompt = PromptTemplate.from_template(""" 
            You are an expert government compliance officer evaluating agricultural scheme eligibility. 
            Evaluate the following User Data against the Scheme Rules. 
            Convert fuzzy regional measurements (like acres or bighas) to hectares if necessary to evaluate 
rules. 
            1 Hectare = 2.47 Acres. 
             
            Scheme Rules: {rules} 
            User Data: {user_data} 
             
            Provide a confidence score between 0.0 and 1.0 based on how clear the user's data is. 
            Return ONLY a valid JSON object matching this exact schema: 
            {{ 
                "eligible": boolean, 
                "missing_fields": ["array", "of", "missing", "keys"], 
                "reason": "Detailed explanation of the decision", 
                "confidence_score": float 
            }} 
        """) 
         
        chain = prompt | self.llm 
        response = await chain.ainvoke({ 
            "rules": json.dumps(scheme_rules),  
            "user_data": json.dumps(user_data) 
        }) 
         
        try: 
            # Strip markdown formatting often returned by generative models 
            raw_json = response.content.replace("```json", "").replace("```", "").strip() 
            result = json.loads(raw_json) 
            return result 
        except json.JSONDecodeError as e: 
            logger.error(f"LLM generated invalid JSON: {str(e)}. Content: {response.content}") 
            return { 
                "eligible": False,  
                "missing_fields":,  
                "reason": "System error during semantic eligibility parsing.", 
                "confidence_score": 0.0 
            } 
 
    async def check_eligibility( 
        self,  
        user_data: Dict[str, Any],  
        scheme_rules: Dict[str, Any] 
    ) -> Dict[str, Any]: 
        """Main entry point for the hybrid engine.""" 
        handled, eligible, missing, reason, confidence = 
self._evaluate_deterministic_rules(user_data, scheme_rules) 
         
        if handled: 
            logger.info("Eligibility conclusively handled by deterministic rule engine.") 
            return { 
                "eligible": eligible, 
                "missing_fields": missing, 
                "reason": reason, 
                "confidence_score": confidence 
            } 
             
        logger.info("Fuzzy data detected. Falling back to LLM semantic evaluation.") 
        return await self._evaluate_llm_fallback(user_data, scheme_rules) 
 
Part 4: Voice-First Interaction Pipeline with Sarvam AI 
To serve the rural agricultural demographic effectively, the system must accommodate 
regional dialects, ambient noise, and code-mixing (e.g., Hindi interspersed with English 
technical terms). Sarvam AI’s saaras:v3 Speech-to-Text (STT) and bulbul:v3 Text-to-Speech 
(TTS) APIs are uniquely optimized for these Indian linguistic nuances, outperforming generic 
models on localized phonetics.13 
The voice service acts as the primary sensory input/output layer for the LangGraph agent. It 
manages asynchronous HTTP requests to Sarvam's REST endpoints, converting raw audio bytes 
to text, and synthesizing the agent's textual responses back into high-fidelity audio arrays.15 
STT, Extraction, and TTS Implementation 
 
Python 
 
 
# services/voice_service.py 
import aiohttp 
import base64 
import logging 
from typing import Optional 
 
logger = logging.getLogger(__name__) 
 
class SarvamVoiceService: 
    """ 
    Integrates Sarvam AI endpoints for Indic language Speech-to-Text  
    and Text-to-Speech processing. 
    """ 
    def __init__(self, api_key: str): 
        self.api_key = api_key 
        self.stt_url = "https://api.sarvam.ai/speech-to-text" 
        self.tts_url = "https://api.sarvam.ai/text-to-speech" 
        self.headers = { 
            "api-subscription-key": self.api_key, 
            "Content-Type": "application/json" 
        } 
 
    async def speech_to_text(self, audio_bytes: bytes, language_code: str = "hi-IN") -> str: 
        """ 
        Transcribes incoming audio using Saaras v3. 
        The model inherently handles code-mixing and background noise. 
        """ 
        # Aiohttp requires FormData for multipart/form-data uploads 
        form_data = aiohttp.FormData() 
        form_data.add_field('file', audio_bytes, filename='input_audio.wav', content_type='audio/wav') 
        form_data.add_field('model', 'saaras:v3') 
        form_data.add_field('mode', 'transcribe') 
         
        # Override headers; multipart sets its own boundary/content-type 
        stt_headers = {"api-subscription-key": self.api_key} 
 
        async with aiohttp.ClientSession(headers=stt_headers) as session: 
            try: 
                async with session.post(self.stt_url, data=form_data, timeout=15) as response: 
                    if response.status == 200: 
                        data = await response.json() 
                        transcript = data.get("transcript", "").strip() 
                        logger.info(f"STT Transcript generated successfully.") 
                        return transcript 
                    else: 
                        error_text = await response.text() 
                        logger.error(f"Sarvam STT HTTP Error {response.status}: {error_text}") 
                        return "" 
            except Exception as e: 
                logger.error(f"STT Network Exception: {str(e)}") 
                return "" 
 
    async def text_to_speech( 
        self,  
        text: str,  
        target_language: str = "hi-IN",  
        speaker: str = "shubh" 
    ) -> Optional[bytes]: 
        """ 
        Synthesizes text to speech using Bulbul v3. 
        Supports granular controls over pitch, pace, and sample rate. 
        """ 
        payload = { 
            "inputs": [text], 
            "target_language_code": target_language, 
            "speaker": speaker, 
            "pitch": 0, 
            "pace": 1.1,  # Slightly faster pace for conversational cadence 
            "loudness": 1.2, 
            "speech_sample_rate": 24000, 
            "model": "bulbul:v3" 
        } 
 
        async with aiohttp.ClientSession(headers=self.headers) as session: 
            try: 
                async with session.post(self.tts_url, json=payload, timeout=10) as response: 
                    if response.status == 200: 
                        data = await response.json() 
                        # Sarvam returns a list of base64 encoded audio strings corresponding to inputs 
                        audio_b64 = data.get("audios",) 
                        return base64.b64decode(audio_b64) 
                    else: 
                        error_text = await response.text() 
                        logger.error(f"Sarvam TTS HTTP Error {response.status}: {error_text}") 
                        return None 
            except Exception as e: 
                logger.error(f"TTS Network Exception: {str(e)}") 
                return None 
 
Part 5: LangGraph Conversational Data Collection 
Engine 
The crux of the multi-agent system resides in conversational orchestration. While standard 
prompt-chaining is susceptible to derailment, LangGraph utilizes a state machine paradigm, 
enabling durable execution, cyclical flows, and precise control over conversational routing.16 
The state machine manages an AgentState object containing the missing_fields list. Rather 
than overwhelming the user by requesting all missing information simultaneously, the graph 
iteratively loops through a next_question node, prompting the user for exactly one parameter 
per turn until the state is fully resolved.18 
State Definition, Nodes, and Conditional Routing

# 5. Define Edge Routing Logic 
def routing_logic(state: AgentState) -> str: 
    """Evaluates the state after eligibility check to route the graph.""" 
    if state["current_step"] == "rejected": 
        return "rejection_end" 
    if state["missing_fields"]: 
        return "ask_question" 
    return "generate_form" 
 
Part 6: Automated Form Generation with PyMuPDF 
Once the conversational engine resolves all missing variables and confirms eligibility, the 
pipeline transitions to the physical form fulfillment stage. Official templates (e.g., the standard 
PM-Kisan offline application) are programmatically filled mapping the populated JSON state 
dictionary to the PDF widgets. 
PyMuPDF (fitz) is utilized because of its high-performance widget modification capabilities and 
its ability to robustly "flatten" interactive PDFs.19 Flattening is a critical security requirement; by 
rendering the document into static images embedded in a new PDF shell, post-generation 
tampering of banking or identification details is rendered impossible.20 
 
Python 
 
 
# services/form_filler.py 
import fitz  # PyMuPDF 
import logging 
import os 
from typing import Dict, Any 
 
logger = logging.getLogger(__name__) 
 
class PDFFormFiller: 
    """ 
    Automates the insertion of user state dictionaries into standardized  
    AcroForm PDF templates, securing the output via flattening. 
    """ 
    def __init__(self, template_dir: str = "templates/"): 
        self.template_dir = template_dir 
 
    def generate_application_form(self, user_data: Dict[str, Any], scheme_id: str, output_path: str) -> str: 
        """ 
        Maps JSON data to PDF widgets, updates values, and flattens the document. 
        """ 
        template_path = os.path.join(self.template_dir, f"{scheme_id}_template.pdf") 
         
        if not os.path.exists(template_path): 
            logger.error(f"Template for {scheme_id} not found at {template_path}") 
            raise FileNotFoundError(f"Missing template for {scheme_id}") 
 
        try: 
            # 1. Open the interactive PDF template 
            doc = fitz.open(template_path) 
             
            # Field mapping protocol: Maps PDF AcroForm Widget Names to internal JSON Keys 
            # In a full deployment, this mapping resides in the database per scheme 
            field_mapping = { 
                "Text_FarmerName": "farmer_name", 
                "Text_Aadhaar": "aadhaar_number", 
                "Text_BankAcc": "bank_account", 
                "Text_IFSC": "ifsc_code", 
                "Text_LandArea": "land_area", 
                "Dropdown_Category": "farmer_category" 
            } 
 
            # 2. Iterate through document pages and extract all interactive widgets 
            for page in doc: 
                widgets = page.widgets() 
                for widget in widgets: 
                    field_name = widget.field_name 
                     
                    if field_name in field_mapping: 
                        json_key = field_mapping[field_name] 
                        if json_key in user_data: 
                            # 3. Mutate the widget value 
                            widget.field_value = str(user_data[json_key]) 
                            widget.update() 
             
            # 4. Flattening Protocol: Render the PDF to image arrays to lock fields 
            imaged_doc = fitz.open() 
            for page in doc: 
                # 300 DPI provides standard print-ready resolution 
                pix = page.get_pixmap(dpi=300) 
                img_page = imaged_doc.new_page(width=page.rect.width, height=page.rect.height) 
                img_page.insert_image(page.rect, pixmap=pix) 
 
            # Save the secured, deflated output 
            imaged_doc.save(output_path, deflate=True) 
            doc.close() 
            imaged_doc.close() 
             
            logger.info(f"Successfully generated flattened PDF at {output_path}") 
            return output_path 
             
        except Exception as e: 
            logger.error(f"Failed to generate form: {str(e)}") 
            raise e 
 
Part 7: Status Tracking and Human-in-the-Loop 
Validation 
Asynchronous processing workflows spanning multiple interactions require rigorous state 
tracking. Furthermore, GovTech systems necessitate explicit human validation before 
submitting critical forms bearing Aadhaar and banking details to government API gateways. 
LangGraph natively supports Human-In-The-Loop (HITL) workflows via its dynamic interrupt() 
method. When invoked, interrupt() pauses the graph's execution, serializes the current state 
into the persistent Checkpointer, and waits indefinitely until an external system (via an API 
command) signals it to resume.21 
HITL Node and FastAPI Endpoints 
Table 2 highlights the strict state enums the system maintains to ensure synchronous 
awareness between the UI, the Voice Interface, and the LangGraph backend.

    """ 
    Retrieves the exact execution state of the conversation graph. 
    Useful for frontends polling for progress. 
    """

        config = {"configurable": {"thread_id": thread_id}}

            raise HTTPException(status_code=404, detail="Thread not found")

        current_state = state_snapshot.values 
         
        return { 
            "current_step": current_state.get("current_step"), 
            "collected_data": current_state.get("collected_data", {}),

            "form_pdf_path": current_state.get("form_pdf_path") 
        } 
    except Exception as e: 
        raise HTTPException(status_code=500, detail=str(e)) 
 
@app.post("/confirm") 
async def confirm_submission(req: ConfirmRequest): 
    """ 
    Resumes a LangGraph thread that is paused at an interrupt(). 
    Injects the user's boolean approval decision. 
    """ 
    config = {"configurable": {"thread_id": req.thread_id}} 
     
    try: 
        # Use the Command object to inject data and resume the graph 
        from langgraph.types import Command 
         
        command = Command(resume={"approved": req.approved}) 
         
        # Invoke the graph with the resume command 
        output_state = await orchestrator.app.ainvoke(command, config=config) 
         
        return { 
            "status": "resumed", 
            "final_step": output_state.get("current_step") 
        } 
    except Exception as e: 
        raise HTTPException(status_code=500, detail=f"Failed to resume graph: {str(e)}")

The pipeline module synthesizes the entire architecture. It acts as the central router between

# pipeline/orchestration.py 
import uuid 
import base64 
import logging 
from typing import Dict, Any 
 
from langgraph.graph import StateGraph, START, END 
from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver 
from psycopg_pool import AsyncConnectionPool 
 
from services.voice_service import SarvamVoiceService 
from services.form_filler import PDFFormFiller 
from services.retrieval_service import SchemeRetriever 
from agents.scheme_agent import ( 
    AgentState, extract_entities_node, eligibility_check_node,  
    conversational_prompt_node, human_validation_node,  
    submit_to_portal_node, routing_logic 
) 
 
logger = logging.getLogger(__name__) 
 
class SchemeOrchestrator: 
    """ 
    Central pipeline managing the STT -> Graph -> TTS flow. 
    Utilizes AsyncPostgresSaver for durable execution. 
    """ 
    def __init__(self, db_connection_string: str, sarvam_key: str): 
        self.voice_service = SarvamVoiceService(sarvam_key) 
        self.form_filler = PDFFormFiller("templates/") 
        self.retriever = SchemeRetriever(db_connection_string) 
        self.pool = AsyncConnectionPool(conninfo=db_connection_string, max_size=20) 
         
        self.app = self._build_graph() 
 
    def _build_graph(self): 
        """Constructs the LangGraph directed cyclic graph.""" 
        workflow = StateGraph(AgentState) 
         
        # Add Execution Nodes 
        workflow.add_node("extract_entities", extract_entities_node) 
        workflow.add_node("check_eligibility", eligibility_check_node) 
        workflow.add_node("ask_question", conversational_prompt_node) 
        workflow.add_node("generate_form", self._form_generation_wrapper) 
        workflow.add_node("human_validation", human_validation_node) 
        workflow.add_node("submit_to_portal", submit_to_portal_node) 
         
        # Construct Edges and Routing 
        workflow.add_edge(START, "extract_entities") 
        workflow.add_edge("extract_entities", "check_eligibility") 
         
        # Conditional Router branching based on eligibility and missing fields 
        workflow.add_conditional_edges("check_eligibility", routing_logic, { 
            "ask_question": "ask_question", 
            "generate_form": "generate_form", 
            "rejection_end": END 
        }) 
         
        # When a question is generated, execution ends for the turn (awaiting user response) 
        workflow.add_edge("ask_question", END)  
        workflow.add_edge("generate_form", "human_validation") 
        workflow.add_edge("submit_to_portal", END) 
         
        # Compile with the Checkpointer for state persistence 
        checkpointer = AsyncPostgresSaver(self.pool) 
        return workflow.compile(checkpointer=checkpointer) 
 
    async def _form_generation_wrapper(self, state: AgentState) -> Dict[str, Any]: 
        """Wrapper node bridging the graph state to the PDF filler service.""" 
        scheme_id = state.get("target_scheme", {}).get("scheme_id", "default") 
        output_file = f"outputs/form_{uuid.uuid4().hex[:8]}.pdf" 
         
        self.form_filler.generate_application_form( 
            user_data=state["collected_data"],  
            scheme_id=scheme_id, 
            output_path=output_file 
        ) 
        return {"form_pdf_path": output_file, "current_step": "form_generated"} 
 
    async def run_scheme_agent(self, audio_input: bytes, user_context: Dict[str, Any]) -> Dict[str, Any]: 
        """ 
        Primary execution function. Converts audio to intent, updates state,  
        progresses the graph, and responds via voice. 
        """ 
        thread_id = user_context.get("session_id", str(uuid.uuid4())) 
        config = {"configurable": {"thread_id": thread_id}} 
        language = user_context.get("language_pref", "hi-IN") 
         
        # 1. Voice to Text Translation (Sarvam AI) 
        user_text = await self.voice_service.speech_to_text(audio_input, language_code=language) 
        if not user_text: 
            return {"error": "Audio parsing failed. Please try speaking clearly."} 
 
        logger.info(f"User Input ({thread_id}): {user_text}") 
 
        # 2. Inject Context and Trigger Graph Execution 
        input_state = { 
            "messages": [HumanMessage(content=user_text)], 
            "language_pref": language, 
            "user_profile": user_context.get("profile", {}) 
        } 
         
        # RAG Retrieval is injected if a scheme isn't already targeted 
        current_snapshot = await self.app.checkpointer.aget(config) 
        if not current_snapshot or not current_snapshot.values.get("target_scheme"): 
            user_state = user_context.get("profile", {}).get("state", "Central") 
            user_land = user_context.get("profile", {}).get("land_area", 0.0) 
             
            docs = await self.retriever.retrieve_relevant_schemes(user_text, user_state, 
float(user_land)) 
            if docs: 
                # Retrieve the JSON rules mapped to the top vector hit 
                target_scheme_json = docs.metadata 
                input_state["target_scheme"] = target_scheme_json 
                input_state["current_step"] = "scheme_identified" 
         
        # Execute the State Machine 
        output_state = await self.app.ainvoke(input_state, config=config) 
         
        # 3. Handle Interrupts (Human Validation) 
        interrupts = output_state.get("interrupts",) 
        if interrupts: 
            return { 
                "status": "awaiting_user_confirmation", 
                "validation_data": interrupts 
            } 
             
        # 4. Text to Speech Response 
        ai_message = output_state["messages"][-1].content 
        audio_response = await self.voice_service.text_to_speech(ai_message, 
target_language=language) 
 
        return { 
            "status": output_state.get("current_step", "in_progress"), 
            "text_response": ai_message, 
            "audio_response_b64": base64.b64encode(audio_response).decode('utf-8') if 
audio_response else None 
        }
